# PARADE 论文与 Parent-Child RAG 项目对照稿设计

## 1. 目标

围绕论文 *PARADE: Passage Representation Aggregation for Document Reranking*，维护一组面向 RAG/LLM 应用算法实习求职的公开材料。

材料不以完整复述论文为目标，而是展示一条可验证的项目叙事：从 Parent-Child RAG 中的真实排序问题出发，引入 passage-to-document aggregation 的研究框架，通过受控消融验证适用于当前系统的聚合策略，并说明论文结论迁移到工程系统时的边界。

用户不需要预先补齐信息检索术语或独立通读英文论文。Codex 负责原文阅读、术语解释、项目映射和文档维护；用户只需对事实归属、个人理解和公开范围做一次轻量复核。

## 2. 目标读者与使用场景

主要读者是 RAG/LLM 应用算法岗位的招聘方和面试官，也兼顾尚无系统信息检索背景的技术读者。

材料需要支持三种使用场景：

1. 在公开仓库中展示项目的问题分析、实验设计和研究迁移能力；
2. 从完整材料中提炼简历 bullet；
3. 支持 30 秒概述、2 分钟介绍和进一步技术追问。

## 3. 产出文件

### 3.1 公开案例主稿

文件：`docs/research/parade/case-study.md`

主稿采用项目问题优先的结构：

1. **摘要**：用 150-200 字说明问题、实验、结果和最终决策；
2. **项目中的真实问题**：解释 child、parent、source，以及检索、排序和消费单位不一致造成的重复占位；
3. **PARADE 的问题框架**：介绍 MaxP、score aggregation、representation aggregation，以及单点相关性和分散证据的区别；
4. **论文与项目对照**：列出可以映射的概念和不能直接类比的边界；
5. **受控实验**：说明 PC0、PC1、PC2、H1 分别改变了什么、冻结了什么；
6. **结果与解释**：展示 PC2 的提升、H1 的覆盖与 MRR 权衡，并讨论重复证据、overlap 和长文档偏置；
7. **工程决策**：说明为何采用 PC2、为何不围绕失败 case 调参，以及当前未发布线上索引的边界；
8. **岗位相关能力**：总结 retrieval、reranking、evaluation、受控消融和工程判断；
9. **局限与未来工作**：记录 evidence-level labels、真实 prompt budget 和答案质量评测等后续方向。

正文中的术语在首次出现时用一句话解释。主稿不假设读者已经具备完整的信息检索知识。

### 3.2 PARADE 精读笔记

文件：`docs/research/parade/reading-notes.md`

精读笔记作为研究底稿，不追求连续文章形式。对论文各部分固定记录：

- 作者要解决的问题；
- 关键术语的中文解释；
- 模型、公式或实验的直观含义；
- 实验实际支持的结论；
- 不能从实验推出的结论；
- 与 PC2/H1 的联系；
- 可引用到公开主稿的原文位置。

笔记附带最小必要术语表，包括 ad-hoc retrieval、first-stage retrieval、reranking、relevance signal、MaxP、score aggregation、representation aggregation、MAP、MRR 和 nDCG。术语说明以读懂 PARADE 和当前项目为限，不扩展成完整的信息检索教程。

### 3.3 面试指南

文件：`docs/research/parade/interview-guide.md`

面试指南包含：

- 2-3 条简历 bullet；
- 30 秒项目概述；
- 2 分钟完整介绍；
- 10 个高概率追问；
- 每个问题的简洁回答和深入回答；
- 容易过度表述的红线。

三份材料以公开案例主稿为叙事口径，以项目实验记录和论文原文为事实来源，避免出现数字、结论或边界不一致。

## 4. 事实来源与映射边界

项目事实和实验数字以 `docs/experiments/retrieval/results.md` 为主要来源，并与相关实验设计、评测代码和 README 交叉核对。

论文结论以 PARADE 原文为主要来源。关键判断需要记录可追溯的章节、图表或实验位置，公开主稿使用规范引用。

对照稿必须明确以下边界：

- PARADE 研究 passage-to-document reranking；当前项目研究 child-to-parent-to-source 两级聚合；
- PARADE 的主要方法是有监督的 passage representation aggregation；PC2/H1 是 hybrid retrieval 和 child-level RRF 之后的无训练分数聚合；
- PC2 与 MaxP 只有思想和聚合假设上的对应，PC2 还包含 source 去重和消费单位对齐；
- H1 是有限 Top-k score aggregation，不等同于 PARADE 的 Transformer 或 CNN representation aggregator；
- 当前实验只能支持离线 retrieval 排序结论，不能证明最终答案质量提升；
- 对外表述只能说“参考 PARADE 的问题框架解释和设计聚合实验”，不能说“实现了 PARADE”。

## 5. 维护流程

1. 阅读 PARADE 原文，先建立精读笔记、术语表和原文证据定位；
2. 将论文的任务设定、聚合方法和实验结论与 PC0/PC1/PC2/H1 逐项对照；
3. 编写公开案例主稿，只纳入与项目叙事直接相关且证据充分的内容；
4. 从主稿提炼面试指南，确保精简表达不改变事实口径；
5. 复核引用、数字、术语解释、方法映射和迁移边界；
6. 由用户进行一次轻量复核，确认工作归属、个人可理解性和公开范围；
7. 根据复核结果做最终修订。

后续项目实验或默认检索行为发生变化时，先更新 `docs/experiments/retrieval/results.md`，再同步对照主稿和面试指南。论文精读笔记只在新发现影响解释或引用时更新。

## 6. 用户参与方式

用户无需独立通读英文论文，也无需先学习完整的信息检索课程。用户只负责确认：

1. 文档描述的工作是否确实属于本人完成的项目工作；
2. 面试表述是否能在短时间熟悉后自然讲出；
3. 文档是否包含不希望公开的信息。

若用户无法解释某个概念，优先简化或补充直观解释，不把该概念直接包装成用户已经掌握的能力。

## 7. 质量标准

- 每个关键项目数字都能回溯到现有实验记录；
- 每个关键论文判断都能回溯到论文原文；
- 不把论文结论写成对当前项目的直接证明；
- 不把 retrieval 指标写成最终 RAG 回答质量；
- 不虚构用户独立阅读、复现或实现 PARADE 的经历；
- 主稿能被没有系统 IR 背景的 RAG 技术读者理解；
- 面试材料中的内容必须可以由用户在短时间熟悉后亲自解释；
- 三份材料在实验数字、采用决策和工程边界上保持一致；
- 公开稿不包含密钥、私有配置、个人信息或未经确认的内部材料。

## 8. 验收方式

文档验收采用静态核查，不运行新的检索实验：

1. 对项目数字逐项比对 `docs/experiments/retrieval/results.md`；
2. 对论文结论逐项比对 PARADE 原文；
3. 搜索不允许的过度表述，例如“实现 PARADE”“证明答案质量提升”；
4. 检查三份材料中的 PC2/H1 数字和采用结论是否一致；
5. 检查所有外部引用是否可访问、标题和作者信息是否准确；
6. 由用户完成一次事实、理解和公开范围复核。

## 9. 范围边界

本轮只创建和维护上述三份文档，不修改检索代码，不运行新实验，不调整 H1 权重，不发布生产索引，也不新增端到端答案质量指标。

如果后续需要实现 PARADE 类学习式聚合、补充 evidence benchmark 或修改线上检索链路，应另行设计和评估，不并入本次短期求职材料整理。
